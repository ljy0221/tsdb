## 해석

### 1. WAL DSYNC가 쓰기 경로의 물리 상한
`memTableOnly` : `memTablePlusWal` = 9,465,502 : 389 ≈ **24,300:1**.
MemTable put 자체는 ~100ns 수준이지만, WAL append마다 fsync(DSYNC)가 강제되며
한 번의 fsync에 **≈ 2.57 ms** (1000/389)가 소요된다. WSL2 ext4 + SSD에서 합리적 수치.
즉 이 환경에서 **durable 쓰기의 물리 상한은 ~400 ops/s**이며, 코드 최적화로는 못 넘는다.

### 2. Phase 1 완료 기준 충족 (정합성·성능 양 측면)
Phase 1 완료 기준은 "10만 건 write → flush → read 정합성". `endToEndFlush`는 정확히 그
경로를 측정하며 **3.27 M ops/s**가 나온다. 즉 100,000건 put + flush 1회를 **30.6ms**에
완료한다. 병합 읽기(`MergingIterator`)는 정합성 테스트로 검증 완료.

### 3. LSM의 가치가 수치로 드러남
`endToEndFlush`는 `memTableOnly`의 35%로 살아남는다. SSTable 쓰기 경로에 fsync가 있지만
**10만 건당 1회**로 amortize되기 때문이다. WAL처럼 put당 fsync를 강제하는 구조와
비교하면 **약 8,400× throughput 이득**. LSM이 해결하는 문제의 본질이
"fsync 횟수를 O(1)/batch로 만들기"임이 데이터로 확인된다.

### 4. Compaction 설계 기준
Compaction은 **쓰기 경로에 fsync를 추가하지 않는 구조**여야 한다. 쓰기 경로의 지배
비용이 WAL fsync(2.5ms)이므로, compaction이 쓰기와 동기화되면 ops/s가 추가로 반토막.
읽기 경로의 SSTable fan-out을 줄여 read amplification만 공략하는 방향이 타당하다.
