## 해석 — 수치가 말해주는 것

### 1. 조건 3의 `ratio ≈ 1.02×`는 "파일 내부 O(n) 완전 지배"의 결정적 증거

`files=1, RECENT` 에서 selectivity가 0.01이든 1.0이든 p99가 거의 같다 (21.07ms vs 21.55ms).
이는 현재 구현이 **selectivity를 전혀 활용하지 못한다**는 뜻이다. `MergingIterator`가
`r.iterator()` 로 파일 전체를 iterator 화하고, consumer 단에서 `< startTs` 를
단순 스킵(continue) 하기 때문에 시작점을 건너뛸 수 없다. 1% 쿼리든 100% 쿼리든
**정확히 같은 양의 바이트를 mmap에서 읽는다**.

spec이 이론 상한을 100×로 가정하고 "20× 미만이면 실패"로 잡았는데, 실측은
**1× 근처**로 나왔다. 즉 현재 구현에서 selectivity는 무효 파라미터에 가깝다.

### 2. files=4 가 files=1 보다 빠른 이유 — `SSTableMeta.overlaps()` 가 해내는 일

거의 모든 비교 쌍에서 `files=4 < files=1`:
- `UNIFORM/0.01`: files=4 p99 5.8ms vs files=1 p99 17.3ms (**2.97× 빠름**)
- `RECENT/0.01`: files=4 p99 6.7ms vs files=1 p99 21.1ms (**3.14× 빠름**)

이는 spec이 예상한 방향과 일치한다: `SSTableMeta.overlaps()` 로 4개 중 최대 1개
파일만 열고 나머지 3개는 건너뛴다. **파일 단위 프루닝은 잘 작동하고 있다.**

역설: 이 결과는 "compaction이 읽기 성능을 **해친다**"고 해석될 수 있다.
Phase 1에서 compaction이 파일 수를 줄이는 것을 "좋은 일"로 전제했는데,
읽기 관점에서는 파일 수가 줄면 **파일 단위 프루닝이 먹을 여지가 줄어든다**.
이 긴장은 Phase 2 인덱스 도입으로 해소된다 — 파일 내부에도 스킵 가능 구조가
생기면 큰 파일 하나가 작은 파일 여러 개보다 나빠지지 않는다.

### 3. RECENT가 UNIFORM보다 느린 예상 밖 방향

- `files=1/0.01`: UNIFORM 17.3ms < RECENT 21.1ms
- `files=1/1.0`: UNIFORM 20.5ms < RECENT 21.5ms

나는 "RECENT는 같은 tail을 반복 스캔하니 page cache 이점으로 더 빠를 것"이라
가정했는데, 반대로 나왔다. 원인 추정: `MergingIterator` 는 `> endTs` 에서 break
하는데, RECENT 쿼리는 `endTs == dataMax` 이라 **break가 절대 발동하지 않고
무조건 끝까지 간다**. UNIFORM은 평균적으로 중간 어딘가에 `endTs` 가 있어
`break` 덕에 덜 읽는다.

이 관찰은 인덱스 설계에서 **"끝에서부터 읽는 역방향 스캔"을 고려할 근거**로
쓰일 수 있다 (TSDB의 실제 쿼리 패턴이 "최근 N분"이므로). 단, 우선순위는
낮게. 먼저 "시작점을 건너뛰는 sparse index" 가 훨씬 큰 레버.

### 4. 조건 1과 조건 2의 갭 — 인덱스의 예상 이득 크기

| 조건 | p99 | 개선 타겟 (5ms 기준) | 개선 배율 |
|------|-----|---------------------|-----------|
| files=4 | 6.7 ms | ≤ 5 ms | 1.34× 필요 |
| files=1 | 21.1 ms | ≤ 5 ms | 4.21× 필요 |

`files=1` 의 21ms를 5ms 아래로 낮추려면 **4× 이상 개선**이 필요하다.
1M 엔트리 이진 탐색은 이론상 `log₂(1M) ≈ 20 step`으로 시작점 찾기,
그 뒤 sparse index 블록 경계 하나만 읽으면 되므로 **1% 쿼리는 10,000 entry 읽기
+ 20 step 이진 탐색** 으로 끝난다. 이론상 100× 근처 개선이 가능하며, 실제로는
캐시·MergingIterator 힙 비용 등이 있어 **5~20× 범위**가 현실적이다.

인덱스 ADR의 완료 기준(spec §4 후반):
- **절대 기준**: `p99(files=4, RECENT, 0.01) < 5 ms` + `p99(files=1, RECENT, 0.01) < 5 ms`
- **상대 기준**: `files=1, RECENT, 0.01` baseline 21ms 대비 **5× 개선**
- 둘 중 **더 관대한 쪽** 달성으로 완료 판정.

## 벤치 자체의 한계 (다음 단계에 기억)

- **1M 엔트리 / 16 MB는 WSL2 page cache에 전부 들어간다.** 즉 이 수치는
  "캐시 히트 상태 CPU 바운드"의 스냅샷. 디스크 바운드 시나리오(100M+)는 별도.
- **파일 간 시간 구간은 비중첩이다.** 실제 워크로드에서는 지연 도착으로 겹침이
  생겨 `overlaps()` 프루닝이 덜 효과적일 수 있다.
- `@Setup(Level.Invocation)` 덕에 warm-cache 편향은 제거되었지만, **동일 fixture를
  반복 재사용**하므로 mmap 상 page-fault 오버헤드는 측정에서 제외된 상태.
